# Onboard

Telegram-бот для обучения сотрудников по роли и последующего тестирования через LLM.

Бот обучает сотрудника по инструкции из файла, задает вопросы по одному, проверяет ответы по материалу и сохраняет результат теста в Postgres.

## Возможности

- выбор роли сотрудника: `auditor` или `operator`;
- обучение только по материалу выбранной роли;
- ответы на вопросы сотрудника только по инструкции;
- тест из JSON-файла выбранной роли;
- подсчет результата и процента правильных ответов;
- статус `Тест сдан` / `Тест не сдан` по порогу `PASSING_SCORE_PERCENT`;
- сохранение результата в таблицу `training_results`.

## Как Работает

1. Пользователь отправляет `/start`.
2. Бот предлагает выбрать роль: `auditor` или `operator`.
3. Бот просит имя сотрудника.
4. Бот предлагает выбрать режим: `Обучение + тест` или `Только тест`.
5. В режиме `Обучение + тест` бот обучает по файлу выбранной роли:
   - `materials/auditor.txt` для аудитора;
   - `materials/operator.txt` для оператора.
6. В режиме `Только тест` бот сразу переходит к тесту из JSON-файла роли.
7. После теста бот считает результат, определяет статус сдачи и сохраняет итог в Postgres.

## Материалы

Материалы лежат в папке `materials/`.

```text
materials/
  auditor.txt
  operator.txt
  tests/
    auditor.json
    operator.json
```

Каждый файл самодостаточный: содержит вводную часть, правила роли, примеры, ошибки и критерии проверки. Файлы оформлены в Markdown, но оставлены с расширением `.txt`, чтобы не менять логику загрузки.

Тестовые вопросы лежат отдельно в `materials/tests/*.json`. Код сам задает вопросы по порядку, проверяет ответы и считает процент. AI подключается только для пояснений по материалу и итоговой обратной связи.

## Как Составлять Материалы Для Обучения

Обучающие материалы лежат в файлах:

```text
materials/auditor.txt
materials/operator.txt
```

Каждый файл должен быть самостоятельной инструкцией для одной роли. В файл аудитора не нужно подробно добавлять обязанности оператора, а в файл оператора — обязанности аудитора. Если роль должна понимать, что делает другая сторона, добавьте только короткое пояснение.

Материал должен быть оформлен в Markdown. Бот делит обучение на темы по заголовкам второго уровня:

```markdown
## 1. Вводная часть

Текст темы.

## 2. Общие правила

Текст темы.

## 3. Практический алгоритм

Текст темы.
```

Правила оформления:

- каждая тема начинается с `## Номер. Название темы`;
- внутри темы можно использовать подзаголовки `###`;
- списки оформляйте через `-`;
- чек-листы можно оформлять через `- [ ]`;
- одна тема должна объяснять один крупный смысловой блок;
- не дублируйте один и тот же блок в разных темах без необходимости;
- порядок тем должен совпадать с логикой обучения: сначала база, потом алгоритм, потом ошибки, потом спорные ситуации;
- спорные или неочевидные случаи лучше писать отдельным блоком;
- если правило важно для теста, оно должно быть явно написано в материале.

Бот показывает темы по порядку из файла. Если номера идут не подряд, например `## 9` и потом `## 11`, бот все равно пройдет их в порядке расположения в файле.

Хорошая структура темы:

```markdown
## 4. Проверка фото

Оператор проверяет, что фото товара и фото ценника относятся к одной позиции.
Цена, название товара и единица измерения должны читаться.

### Что проверить

- фото товара есть;
- фото ценника есть;
- цена читается;
- ценник относится к нужному товару.

### Когда вернуть в брак

- фото размыто;
- цена не читается;
- ценник относится к другому товару.
```

## Как Составлять Тесты

Тесты лежат в файлах:

```text
materials/tests/auditor.json
materials/tests/operator.json
```

Количество вопросов берется из JSON-файла. Отдельной переменной окружения для количества вопросов нет.

Поддерживаемый формат:

```json
[
  {
    "test_title": "Тест для оператора",
    "project": "Мониторинг цен",
    "recommended_settings": {
      "questions_count": 20,
      "passing_score": 16
    },
    "questions": [
      {
        "question": "Что нужно сделать перед проверкой фото?",
        "type": "single_choice",
        "options": {
          "A": "Сразу переписать цену.",
          "B": "Прочитать название товара и комментарий.",
          "C": "Проверить только цену.",
          "D": "Открыть следующий товар."
        },
        "correct_answers": ["B"],
        "explanation": "Сначала нужно понять условия задания."
      }
    ]
  }
]
```

Поля вопроса:

- `question` — текст вопроса;
- `type` — `single_choice` или `multiple_choice`;
- `options` — варианты ответа от `A` до `E`;
- `correct_answers` — список правильных вариантов;
- `explanation` — короткое пояснение, которое бот покажет после ответа.

Для одного правильного ответа:

```json
"correct_answers": ["B"]
```

Для нескольких правильных ответов:

```json
"correct_answers": ["A", "C", "D"]
```

Правила составления тестов:

- вопрос должен проверять одно конкретное правило;
- правильный ответ должен однозначно следовать из обучающего материала;
- неправильные варианты должны быть правдоподобными, но явно неверными по инструкции;
- не используйте вопросы, на которые нельзя ответить по материалу;
- для `single_choice` указывайте один правильный ответ;
- для `multiple_choice` указывайте два или больше правильных ответа;
- варианты ответа обозначайте только буквами `A`, `B`, `C`, `D`, `E`;
- пояснение должно коротко объяснять правило, а не просто повторять букву ответа.

Если вопрос относится только к одной роли, он должен лежать только в тесте этой роли.

## Переменные Окружения

Создайте `.env` на основе `.env.example` и заполните реальные значения.

```env
BOT_TOKEN=your_telegram_bot_token
DATABASE_URL=postgresql+asyncpg://postgres:postgres@localhost:5432/onboarding
OPENAI_API_KEY=your_openai_api_key
OPENAI_MODEL=gpt-5.4-mini-2026-03-17
OPENAI_BASE_URL=https://api.openai.com/v1

TRAINING_TOPIC=Ценовой мониторинг
TRAINING_AUDITOR_MATERIAL_FILE=./materials/auditor.txt
TRAINING_OPERATOR_MATERIAL_FILE=./materials/operator.txt
TRAINING_AUDITOR_QUIZ_FILE=./materials/tests/auditor.json
TRAINING_OPERATOR_QUIZ_FILE=./materials/tests/operator.json

PASSING_SCORE_PERCENT=80
LOG_LEVEL=INFO
```

Для локального запуска через `python main.py` используйте хост `localhost`.

Для запуска бота внутри Docker Compose используйте хост `db`:

```env
DATABASE_URL=postgresql+asyncpg://postgres:postgres@db:5432/onboarding
```

## Локальный Запуск

Команды для PowerShell из корня проекта:

```powershell
cd C:\Users\Светлана\Downloads\ПРОМПТ_ИНЖЕНЕР\onboard-main

python -m venv .venv
.\.venv\Scripts\Activate.ps1

python -m pip install --upgrade pip
pip install -r requirements.txt

Copy-Item .env.example .env
```

После этого заполните `.env` реальными токенами и запустите:

```powershell
docker compose up -d db
```

Затем:

```powershell
python main.py
```

Если PowerShell запрещает активацию виртуального окружения, выполните:

```powershell
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\.venv\Scripts\Activate.ps1
```

## Запуск Через Docker

1. Заполните `.env`.
2. Запустите:

```powershell
docker compose up --build
```
